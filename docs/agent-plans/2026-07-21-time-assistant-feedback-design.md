# 时间管理助手反馈修正设计

日期：2026-07-21
状态：方案二已由用户选定，待书面设计复核后实施

## 1. 目标与范围

本次修正覆盖四项：

1. 中长期任务在任务卡和历史详情中隐藏工时展示，但内部继续保留 `est`，不改变现有 API、历史快照和数据库契约。
2. 服务端纠正缺少依据的高紧急度，避免未来或中长期任务被模型习惯性归入第一象限。
3. 保持任务守恒和固定精力比例，不为了填满四象限而强制平均分配任务。
4. 报告生成前提供当天时间约束，生成后拒绝与已有任务冲突的建议，并复用现有机制自动重试一次。

不增加日历集成、工作时间设置、账号配置、数据库字段或完整排程算法。

## 2. 中长期工时展示

- 任务数据继续要求并保存 `est`，矩阵、报告和历史快照结构不变。
- 当前任务卡在 `source === "中长期"` 时不渲染 `est` 标签。
- 历史详情在 `source === "中长期"` 时不拼接 `est`，仍展示重要性、紧急度、来源、截止时间、完成标准和下一步。
- 其他来源的任务继续显示工时。
- 手动添加中长期任务仍可填写并保存预估耗时，本次只修正输出展示，不扩大表单和数据契约范围。

## 3. 紧急度确定性纠偏

在任务提取模型输出通过 Schema 和语义校验后，由服务端统一应用以下规则：

1. 截止日期为今天或已经逾期：`urgency` 固定为 `高`。
2. 截止日期明确在明天或未来，或者来源为 `中长期`：
   - 若任务及对应原始目标包含明确紧迫信号，保留模型给出的 `高`；
   - 否则模型给出的 `高` 纠正为 `中`，其他等级保持不变。
3. 截止日期无法解析且来源不是中长期：保留模型判断，避免猜测。

首批明确紧迫信号限定为：`紧急`、`立即`、`马上`、`尽快`、`今天必须`、`今日必须`、`当天交付`、`影响当天交付`、`阻塞`。不使用笼统的“重要”“重点”作为紧迫依据。

期限解析在现有 ISO 日期基础上补充中文相对日期 `今天`、`今日`、`明天`，并保留 Asia/Shanghai 服务端参考日期。该纠偏只改变紧急度，不改变重要性、任务 ID、来源、截止时间或其他业务字段。

矩阵仍按正式契约映射：只有 `高` 才算紧急或重要，四象限精力比例固定为 55/25/15/5；允许任意象限为空。

## 4. 报告时间冲突校验

### 4.1 调度上下文

服务端在调用报告模型前，从当天或已逾期且含明确时间的任务构造 `scheduleContext`：

- `fixedPoints`：任务截止时间点；
- `protectedWindows`：当 `est` 可解析时，以截止时间向前回推得到的保护时间段；
- 每项约束只包含任务 ID、任务名、起止时间和截止时间，不新增持久化数据。

工时优先解析明确的小时、分钟和小时区间；无法可靠解析时只保留截止时间点，不猜测持续时长。

### 4.2 模型约束与返回校验

- `scheduleContext` 随当前任务、矩阵和优先级上下文一起传给报告模型。
- 提示词要求所有明确时间建议避开其他任务的保护窗口和截止点。
- 服务端扫描 `energyRules` 与 `adjustments` 中的明确 `HH:MM-HH:MM` 时间段。
- 若建议时间段与另一项当天任务的保护窗口或截止点重叠，且该文本并非在安排被保护任务本身，则判为 `MODEL_OUTPUT_INVALID`。
- 第一次冲突复用现有报告工作流重试；连续两次冲突返回现有安全错误，不向前端暴露模型原文或内部时间数据。
- 只出现截止时间说明、没有形成建议时间段的文字不误判为排程冲突。

## 5. 测试设计

### 服务端单元测试

- 今天和逾期任务强制高紧急度。
- 明天、未来和中长期任务在无紧迫信号时从高纠正为中。
- 明确紧迫信号允许未来或中长期任务保持高紧急度。
- 无法解析的普通期限保留模型等级。
- 四象限允许为空且任务仍只出现一次，比例仍为 55/25/15/5。
- 报告第一次返回冲突时间段、第二次返回无冲突建议时总共调用两次并成功。
- 连续两次冲突时返回稳定的模型输出错误。
- 只复述截止时间但不建议时间段时不触发冲突。

### API 与前端测试

- 任务提取 API 返回纠偏后的紧急度。
- 报告 API 对冲突结果执行一次重试且不泄漏原始结果。
- 当前任务页隐藏中长期任务工时，同时普通任务仍显示工时。
- 历史详情隐藏中长期任务工时，同时保留截止时间、完成标准和下一步。

### 完成门禁

- 目标测试 RED 后再完成最小实现并转 GREEN。
- 完整 `npm test` 通过。
- `git diff --check`、全局 Git hooks 和最终 `git status` 检查通过。
- 不调用真实付费模型，不读取或提交真实 `.env`。

## 6. 兼容性与风险

- 历史数据、API 响应和数据库无需迁移；仅改变展示和服务端派生判断。
- 紧迫关键词是保守白名单，无法覆盖所有自然语言表达；期限不明时保留模型判断作为回退。
- 冲突检测只覆盖明确时间段和可解析的当天约束，不承诺完整日历排程。
- 该设计解决已发现的 `17:00-18:30` 与 `17:00/18:00` 任务冲突，同时避免引入当前范围之外的日历系统。
