# 阿里云开发模式部署 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. Deployment changes the ECS host, systemd, Nginx, firewall, and security group, so execute only after the server owner confirms the maintenance window and model API cost/data policy.

**Goal:** 将带用户名密码认证、SQLite Session 和已完成历史的 `time` 项目以开发模式部署到已有阿里云 ECS，并在不干扰现有服务的前提下通过公网端口 `8011` 访问。

**Architecture:** Express 同时提供前端静态文件、认证/历史 API 和四步业务 API，项目进程继续固定监听 `127.0.0.1:4174`。`systemd` 使用现有 NVM Node.js `v20.20.2` 常驻运行；SQLite 正式库位于 `/var/lib/time`，部署前唯一一致性备份位于 `/var/backups/time`。Nginx 在 HTTP `8011` 反向代理到 `4174`，公网只放行固定 IP `/32` 到 Nginx。

**Tech Stack:** Alibaba Cloud ECS、Ubuntu、Git、NVM、Node.js 20.20.2、Express 5、SQLite 3、systemd、Nginx

## 1. 当前实际部署基线

以下状态由服务器部署过程确认，并以本节作为后续维护真源：

- `teacher` 项目位于 `/opt/apps/teacher`，使用 `4173`，不得停止、覆盖或抢占该端口。
- `time` 项目位于 `/opt/apps/time`，使用内部端口 `4174`。
- 账号与历史代码的本地验证不代表服务器已更新；实际部署前必须重新执行本文的备份门禁、migration、登录和历史验收。
- 运行时复用服务器已有 NVM Node.js `v20.20.2`。
- Node.js 可执行文件固定为 `/root/.nvm/versions/node/v20.20.2/bin/node`。
- ECS 当前为 Ubuntu 22.04.5 LTS、x86_64、glibc 2.35。`sqlite3@6.0.1` 下载到的缓存预编译文件曾要求 `GLIBC_2.38`，因此每次 `npm ci` 后都必须使用本机 Node.js headers（头文件）从源码重编译 `sqlite3`，再验证动态依赖。
- 两个项目只共享 Node.js runtime（运行时），各自保留独立的项目目录、`.env` 和 `node_modules`，不会共享 npm 依赖。
- 未采用 Conda：服务器创建 Conda 环境时受内存压力和完整历史索引解析影响；这不是网络或磁盘故障。当前 NVM 方案已经满足项目的 Node.js 20 要求，无需为了本项目增加 Swap 或重试 Conda。
- 本文仍保留完整的首次部署、更新、回退和故障排查命令；勾选状态不作为服务器实时状态证明，实时状态以 `systemctl`、`ss` 和 `curl` 输出为准。

使用方式：

- 当前服务器已经部署完成时，不要重新执行首次部署的 Task 1–7，也不要重复创建目录、service 或 Nginx 符号链接。
- 日常拉取代码使用 Task 8，数据库恢复使用 Task 9，代码回退使用 Task 10，状态检查和故障定位使用 Task 11。
- 截图只确认了 NVM Node.js、项目端口和部署选择；Nginx `8011`、安全组及公网访问状态仍应通过本文的 `systemctl`、`ss` 和 `curl` 命令在服务器上实时核对。

## 3. 为什么不能直接开放 4174

当前 `server/index.js` 明确执行：

```js
app.listen(config.port, '127.0.0.1', () => {
  process.stdout.write(`Time assistant listening on http://127.0.0.1:${config.port}\n`);
});
```

所以安全组即使开放 `4174`，公网也不能直连。保留 loopback（回环地址）监听并使用 Nginx 代理，不需要为部署改源码，而且能缩小 Node.js 服务暴露面。

## Task 5: 配置 Nginx 公网入口

**Files:**

- Create: `/etc/nginx/sites-available/time`
- Create: `/etc/nginx/sites-enabled/time`

- [ ] **Step 1: 写入反向代理站点**

```bash
sudo tee /etc/nginx/sites-available/time > /dev/null <<'EOF'
server {
    listen 8011;
    listen [::]:8011;
    server_name _;

    location / {
        proxy_pass http://127.0.0.1:4174;
        proxy_http_version 1.1;
        proxy_set_header Host $http_host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_connect_timeout 10s;
        proxy_send_timeout 90s;
        proxy_read_timeout 90s;
        client_max_body_size 128k;
    }
}
EOF
```

Expected: `$http_host` 保留浏览器访问的 `:8011` 端口，使 Express 收到的 `Host` 与浏览器 `Origin` 一致，避免 CSRF 同源校验误拒绝。Nginx 对模型的一次重试链路预留 90 秒；请求体上限略高于应用的 `64kb` 限制。

## 12. 本次方案边界

- 这是开发模式部署：源码直接运行，没有前端 build，也没有容器化。
- 服务器实际使用 NVM Node.js `v20.20.2`，不是最初计划的项目内 Conda 环境；共享 Node.js runtime 不等于共享依赖，本项目仍使用自己的 `/opt/apps/time/node_modules`。
- `time.service` 使用 NVM 版本目录的绝对 Node 路径，不依赖交互式 shell 自动加载 `nvm.sh`。将来升级或删除 `v20.20.2` 前，必须先同步修改 service 并验证，避免服务重启后出现 `status=203/EXEC`。
- 当前代码不会自动热重载。修改代码或 `git pull` 后必须执行 `systemctl restart time`；“开发模式”在本项目里表示运行源码，而不是 Node.js watcher。
- `tests/server/start-script.test.js` 是 Windows 批处理脚本测试，当前会直接调用 `cmd.exe`；Ubuntu 服务器测试命令明确排除它，其余服务端测试照常运行。
- 项目使用 SQLite 持久化用户、Session 和已完成历史；四步草稿仍不持久化，刷新页面不会恢复草稿。
- `/api/health` 只证明进程和路由存活，不验证模型 key、模型名称和供应商服务；完整验收需要一次获授权的真实 AI 流程。
- HTTP 明文没有传输加密，用户名、密码、恢复码、Cookie 和历史正文可能被窃听。固定 IP `/32` 白名单只降低暴露面，不能替代 HTTPS；这是项目所有者接受的剩余风险，不得描述为已消除。
- `time.service` 继续使用 `User=root`。root 运行会放大应用或依赖漏洞的影响，可能波及 ECS 上其他 root 可读项目和密钥；该风险已明确接受。
- 同盘单份备份只能应对错误 migration 和部分人工误操作，无法防止整盘故障、实例丢失或入侵者同时删除正式库和备份；该风险已明确接受。
