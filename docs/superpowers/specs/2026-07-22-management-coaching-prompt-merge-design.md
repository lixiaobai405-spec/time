# 时间管理助手管理教练提示词合并设计

## 1. 背景

甲方提供的新提示词强化了“昨天发现问题、今天采取行动、明天优化管理、后天建设能力”的管理教练逻辑，但其 JSON 结构与当前步骤一的 `fields + overall` 契约不兼容，也不包含任务提取、矩阵判定和优先级报告所需字段。

本设计采用零接口改动方案：保留现有四步工作流和全部正式契约，只把甲方提示词中的管理诊断能力合并到步骤一。

## 2. 目标

- 保留步骤一现有输入、输出和 `overall=need_fix` 阻断逻辑。
- 在单栏 PDCA、SMART 检查之外，增加四栏之间的逻辑关联检查。
- 判断“今天”的事项是否包含管理者应重点投入的高价值管理工作。
- 避免机械要求数字、无依据推断管理短板或生成聊天式追问。
- 保证现有前端、服务端、任务提取、矩阵、报告和历史记录无需修改。

## 3. 非目标

- 不采用甲方提示词的 `yesterday_analysis`、`today_focus` 等新 JSON 结构。
- 不增加 `coaching_question`、整体洞察区域或新的前端组件。
- 不修改 API、数据库、账号、历史记录或跨会话记忆。
- 不修改步骤二至步骤四的输入输出契约。
- 不因为某项工作“可能可以授权”就阻止用户进入任务提取。

## 4. 步骤一结构

步骤一依次执行以下分析：

1. 角色与事实边界确认。
2. 单栏完整性检查。
3. 跨四栏逻辑检查。
4. “今天”的管理价值检查。
5. 将问题映射到现有四栏反馈。
6. 按现有规则生成 `fields + overall`。

角色定位调整为：时间管理助手中的管理目标诊断模块，既检查输入是否可执行，也判断四栏能否形成“问题发现—当前行动—管理优化—长期能力建设”的逻辑链。所有判断必须基于用户输入，不得虚构事实。

## 5. 单栏检查规则

| 输入栏 | 检查要求 |
|---|---|
| 昨天 | 包含目标、结果、差距及原因、改进措施，形成复盘闭环。 |
| 今天 | 包含具体、可执行的当前事项，不能只写笼统方向。 |
| 明天 | 包含明确目标、可验证结果和时间节点。 |
| 后天 | 包含发展方向、时间节点和阶段路径。 |

SMART 中的“可衡量”不等于必须包含数字。明确交付物、完成条件、行为结果或可验证状态也可以视为可衡量。模型不得为了满足 SMART 而虚构数量、比例或其他指标。

## 6. 跨四栏逻辑检查

### 6.1 昨天到今天

检查昨天暴露的未解决问题，今天是否存在对应行动。

- 如果昨天明确指出责任人缺失、风险未解决或结果未达成，今天却没有任何承接动作，则在“今天”返回 `warn`。
- 如果今天已经包含确认、处理、登记或跟进动作，则该关系通过。
- 已经完成的复盘事实只能作为背景，不得要求重复执行。

### 6.2 今天到明天

检查今天解决当前问题后，明天是否在适用时包含避免问题重复发生的优化。

- 只有当输入已经暴露重复性管理问题时，才检查流程、机制或协作方式的优化。
- 不强制所有“明天”目标都必须是制度建设；正常的项目交付和阶段推进仍然有效。
- 如果今天只是持续临时催办，明天也没有任何改善责任、流程或协作方式的目标，可在“明天”返回 `warn`。

### 6.3 明天到后天

检查短期优化或阶段目标是否能够支撑长期方向。

- 如果后天具有明确长期方向，但明天完全没有承接动作或阶段目标，可在缺少内容的一栏返回 `warn`。
- 不得强行把两个没有事实关联的目标解释成同一条管理链。

## 7. 管理价值判断

对“今天”增加以下检查：

- 是否包含需要管理者决策、协调、推动或能力建设的事项。
- 是否全部是笼统或低价值事务性工作。
- 当前行动是否支撑昨天的问题或未来目标。

以下情况可以判为 `warn`：

- 今天只有“处理杂事”“回复消息”“整理文件”等笼统事务。
- 今天全部工作与其他三个维度完全无关。
- 昨天存在明确未解决问题，今天却没有任何对应行动。

以下情况不得仅因此判为 `warn`：

- 今天同时包含高价值管理事项和少量事务性任务。
- 某项工作可以授权，但内容具体、合理且有明确期限。
- 模型只是主观认为管理者不应亲自处理。

## 8. 反馈映射

跨栏问题必须映射到现有四个字段，不增加新字段：

| 发现的问题 | 反馈位置 |
|---|---|
| 昨天复盘本身缺环节 | 昨天 |
| 昨天发现问题但今天没有行动 | 今天 |
| 当前行动缺少必要的后续优化 | 明天 |
| 短期目标无法承接长期方向 | 明天或后天，放在实际缺少内容的一栏 |
| 今天全部为笼统事务性工作 | 今天 |

每栏只返回一组简洁的 `issue` 和 `suggestion`。存在多个问题时应合并同类项并优先报告最影响可执行性的缺口。

甲方提示词中的教练式问题必须改为陈述式建议。例如，不输出“哪些工作可以授权？”，而输出“建议区分必须由管理者亲自决策的事项与可授权的执行性工作”。

## 9. 事实边界

- 只依据用户输入分析，不得虚构负责人、期限、数字、客户、事件或管理问题。
- 证据不足时使用“未体现”“尚未说明”，不得断言用户缺乏某种管理能力。
- 建议可以重组或澄清用户已有内容，但不能替用户作出新的业务决定。
- 已完成工作不得重新生成执行建议。
- 空栏仍按现有契约返回示范文本，并明确标注“为示范，请按实际修改”。

## 10. 输出契约

输出结构保持不变：

```json
{
  "fields": [
    {"key": "昨天", "status": "ok|warn", "issue": "", "suggestion": ""},
    {"key": "今天", "status": "ok|warn", "issue": "", "suggestion": ""},
    {"key": "明天", "status": "ok|warn", "issue": "", "suggestion": ""},
    {"key": "后天", "status": "ok|warn", "issue": "", "suggestion": ""}
  ],
  "overall": "pass|need_fix"
}
```

- 任意一栏为 `warn` 时，`overall` 必须为 `need_fix`。
- 四栏全部为 `ok` 时，`overall` 必须为 `pass`。
- `need_fix` 继续阻止任务提取。
- 不增加解释性文字、Markdown 围栏或其他 JSON 字段。

## 11. 验收场景

1. 四栏信息完整且逻辑连贯：四栏均为 `ok`，`overall=pass`。
2. 各栏单独完整，但昨天的问题没有今天行动：今天为 `warn`。
3. 今天包含高价值管理事项和少量事务性任务：不得仅因存在事务性任务判为 `warn`。
4. 明天使用非数字但可验证的完成条件：不得机械要求补充数字。
5. 后天只有方向，没有时间节点或阶段路径：后天为 `warn`。
6. 建议不得新增用户未提供的负责人、日期、数字或业务事实。
7. 输出仍严格匹配现有 `fields + overall` schema。
8. 用户提供的“四栏输入回归测试内容”应倾向于通过，其中昨天未明确的两项负责人由今天 11:30 前的确认任务承接。

## 12. 预计修改范围

- 修改 `prompts/system.md`：只增强步骤一的角色、分析规则和事实边界。
- 修改 `tests/server/prompt-contract.test.js`：确认新规则存在且输出结构不变。
- 修改 `tests/prompt-cases.md`：增加跨栏关联、管理价值和非数字可衡量场景。

不修改前端、API、数据库以及步骤二至步骤四。
