# 每日跟踪来源对账与截止日期标准化设计

## 背景

当前每日跟踪会把当天历史记录中的任务合并到账号当天清单，并保存用户编辑、完成状态和手动删除标记。现有合并逻辑只会追加新来源任务，不会移除来源历史已经删除的任务。因此，用户删除旧历史后，每日跟踪仍保留旧任务，导致最新拆分只有 11 项而清单累计显示 40 多项。

任务拆分允许模型返回“今天”“明天”等相对截止时间。后端虽然可以解析这类文本并计算紧急度，但没有把解析结果回写为具体日期，导致界面仍显示相对文本。

## 已确认需求

1. 每个账号每天只有一份持续使用的每日跟踪清单。
2. 清单汇总当天仍然存在的全部历史记录任务。
3. 删除一条历史记录后，该记录提供的任务必须从每日跟踪中一起删除。
4. 即使来源任务已经在每日跟踪中被用户编辑，删除来源历史时仍然删除该任务。
5. 其他仍存在的历史记录任务不受影响。
6. “今天”“明天”“后天”等可以唯一计算的截止时间转换为北京时间下的具体日期。
7. “尽快”“月底”“近期”等无法唯一确定的期限不得猜测，统一转换为“待确认”。

## 非目标

- 不改变“一账号、一天、一清单”的数据库唯一性。
- 不新增定时任务、消息推送或跨页面实时推送。
- 不修改历史详情内容；历史快照仍保持保存时的数据。
- 不为任务新增来源历史 ID，也不新增数据库 migration。
- 不批量处理非当天历史记录。
- 不改变页面布局或五步流程。

## 方案选择

采用“读取和保存每日跟踪时与当天历史来源重新对账”的方案。

未采用以下方案：

- 删除历史时直接修改每日跟踪 JSON：会让历史模块与每日跟踪存储直接耦合，并增加与自动保存并发冲突的风险。
- 为每项任务保存来源历史 ID：追踪最精确，但需要新字段、migration 和现有数据回填，本次需求不需要这一级复杂度。

## 每日跟踪对账规则

`historyRepository.listTasksCreatedBetween()` 继续提供当天仍存在历史记录中的全部任务。`dailyTrackingService` 在每次 `GET /api/time-management/daily-tracking/today` 和 `PUT /api/time-management/daily-tracking/today` 时，用当前来源任务 ID 集合对已保存快照进行对账。

对账顺序如下：

1. 对当前来源任务按任务 ID 去重。
2. 过滤已保存任务：
   - 任务 ID 仍存在于当前来源集合，并且没有被用户从每日跟踪手动删除：保留。
   - 任务 ID 已不在当前来源集合：删除，即使任务内容曾被编辑。
3. 已保存任务仍存在时，以已保存版本为准，保留用户对名称、截止日期、工时和优先级的编辑。
4. 当前来源中新增、且没有被用户手动删除的任务追加到清单。
5. `tracking` 只保留最终可见任务的完成状态。
6. `removedTaskIds` 只保留仍存在于当前来源集合的任务 ID，清理已经失效的删除标记。
7. 只要发生任务新增、来源失效删除、完成状态清理或删除标记清理，就把 `hasUnpersistedMerge` 设为 `true`。

前端已经在 `hasUnpersistedMerge=true` 时立即触发自动保存，因此用户删除历史后再进入每日跟踪，会先看到已经对账的清单，随后自动把结果写回数据库。

如果每日跟踪正在另一个页面或标签页打开，不要求服务器主动推送变化；该页面下一次读取或保存时也会执行同样的对账，不会重新写入已经删除来源的任务。

当前产品没有“直接在每日跟踪中新建独立任务”的入口。五步流程中手动补充的任务会进入历史快照，因此仍属于该历史来源；删除该历史时，这些任务同样删除。未来若增加每日跟踪独立任务，需要单独引入来源元数据，不能继续只按当前规则推断。

## 截止日期标准化

截止日期标准化由服务端完成，不依赖模型猜测。统一使用 `Asia/Shanghai` 时区。

支持的输入及输出：

| 输入 | 输出 |
|---|---|
| `今天`、`今日` | `YYYY-MM-DD` |
| `今天 18:00`、`今天18:00前` | `YYYY-MM-DD 18:00` |
| `明天` | 次日的 `YYYY-MM-DD` |
| `明天 09:30` | 次日的 `YYYY-MM-DD 09:30` |
| `后天`及带时间形式 | 后两天的具体日期，可附带时间 |
| 合法的 `YYYY-MM-DD` | 保持并标准化为该格式 |
| 合法的 `YYYY-MM-DD HH:mm` 或 `YYYY-MM-DDTHH:mm` | 标准化为 `YYYY-MM-DD HH:mm` |
| 空值、无效日期、`尽快`、`月底`、`近期`等 | `待确认` |

标准化在两个位置复用：

1. AI 拆分结果进入任务列表时立即标准化，保证步骤二首次展示就是具体日期。
2. 每日跟踪对账时标准化当天来源任务和已保存任务，兼容升级前已经保存的相对日期。

紧急度仍使用同一个解析结果计算：

- 今天到期或已到期：高紧急度。
- 明确日期在未来七天内：中紧急度。
- 更远日期：低紧急度。
- 无法确定日期时显示“待确认”，再按现有来源和紧急信号规则判断紧急度。

历史详情不做后台批量改写。新生成并保存的历史会自然包含标准化后的日期；升级前历史只在进入当天每日跟踪时对展示清单进行兼容标准化。

## API 与数据兼容性

现有 API 路径、请求结构和响应结构保持不变：

- `GET /api/time-management/daily-tracking/today`
- `PUT /api/time-management/daily-tracking/today`
- `DELETE /api/time-management/history/:id`

数据库表和 JSON 字段保持不变，不需要 migration。任务仍以 UUID 作为稳定身份。现有前端无需新增字段，只需要接收标准化后的 `due` 值。

## 并发与错误处理

- 每日跟踪保存继续使用 `revision` 乐观锁；对账不绕过冲突检查。
- 保存请求到达时重新读取当天现存历史来源，因此旧页面无法把已删除历史的任务重新写回。
- 历史删除失败时不改变每日跟踪。
- 每日跟踪对账或保存失败时沿用现有安全错误和重试界面，不返回数据库内部信息。
- 任务 ID 同时存在于多条仍保留的历史中时，删除其中一条不会删除该任务，因为它仍在当前来源集合中。

## 测试设计

### 每日跟踪单元测试

- 已保存且编辑过的任务在来源 ID 消失后被删除。
- 仍存在来源中的编辑任务保留用户版本。
- 新来源任务继续追加。
- 同一任务 ID 出现在多条历史中时只保留一项。
- 被来源删除的任务对应完成状态一起清理。
- 已失效的 `removedTaskIds` 一起清理。
- 任一对账变化都会设置 `hasUnpersistedMerge=true`。

### 服务与 API 测试

- 保存两条当天历史并打开每日跟踪后包含全部任务。
- 删除其中一条历史，再次读取每日跟踪时只返回剩余历史任务。
- 保存旧页面快照时不会恢复已经删除来源的任务。
- 账号隔离保持不变。

### 截止日期测试

- 固定北京时间，验证今天、今日、明天、后天及其时间形式。
- 验证跨月、跨年转换。
- 验证显式日期和日期时间标准化。
- 验证无效日期及模糊期限变为“待确认”。
- 验证标准化后原有紧急度规则不回退。

### 回归测试

- 运行完整服务端测试。
- 运行 Playwright 五步流程、历史记录和每日跟踪相关测试。
- 确认现有自动保存、乐观锁、CSRF 和账号隔离行为不变。

## 验收标准

1. 删除历史记录后重新进入每日跟踪，不再出现该历史的任何任务。
2. 被删除来源任务即使曾经编辑或完成，也从任务和完成状态中消失。
3. 其他当天历史任务和用户编辑保持不变。
4. 每日跟踪自动保存对账后的清单，刷新页面不会恢复旧任务。
5. 新拆分任务不显示“今天”“明天”“后天”等相对截止时间。
6. 无法唯一换算的期限显示为“待确认”，系统不猜日期。
7. 不需要数据库 migration，现有账号数据可直接兼容。
