# 参考稿前端一比一复刻设计

## 背景与目标

甲方提供 `时间管理助手 (1).html` 作为前端验收基准。此次改造以参考稿的视觉与现场演示路径为第一优先级，复刻其顶部导航、工作台、五步梳理流程、每日跟踪和历史记录界面，同时保留当前项目已经验收的真实认证、四个 AI 工作流接口、账号隔离和已完成报告历史。

验收目标是：在桌面和窄屏环境下，页面结构、色彩、间距、文案、控件形态、步骤状态和主要交互与参考稿一致；现有服务端安全与数据能力不得退化。

## 范围

### 本期包含

- 参考稿顶部导航：工作台、梳理流程、每日跟踪、历史记录。
- 参考稿登录/注册视觉，并接入现有真实注册、登录、找回、恢复码和退出流程。
- 参考稿工作台的四类投入卡片、到期提醒和入口操作。
- 五步梳理流程：事务填写、AI 拆解确认、时间分布诊断、优先级排序、优化报告。
- 参考稿任务表格、SMART 缺项提示、手动新增与删除交互。
- 参考稿时间分布条形图、目标区间和诊断说明。
- 参考稿四象限矩阵和优化报告卡片。
- 参考稿每日跟踪、完成时间登记、结束当日并滚动，以及当前会话的每日记录。
- 现有账号报告历史的列表、详情、删除和保存失败重试能力。
- 桌面与窄屏响应式、键盘操作、动态状态语义和无横向溢出回归。

### 本期不包含

- 不新增数据库表、持久化字段或服务端接口。
- 不修改现有 AI 工作流请求/响应契约。
- 不把任务草稿、每日完成状态、滚动结果或每日分布写入 localStorage、sessionStorage、Cookie 或 SQLite。
- 不复制参考稿中的模拟账号、预填密码、演示种子任务或不安全的内存假登录。
- 不改变报告历史的账号隔离、幂等保存、分页和删除规则。
- 不承诺自动排期、团队容量、跨时区协作或生产级日历能力。

## 设计原则

1. 参考 HTML 是视觉和页面交互真源；现有项目是数据、安全和业务契约真源。
2. 优先复用参考稿的 CSS 数值、DOM 分组和文案，不做额外品牌再设计。
3. 前端新增能力只作用于当前 JavaScript 状态树；刷新或退出后清空。
4. 任务、矩阵和报告仍由现有 API 响应驱动，不以参考稿中的本地解析器替代模型调用。
5. 所有下游结果继续遵守失效规则：目标变化清空任务以后结果，任务变化清空矩阵和报告。

## 信息架构与页面

### 顶部导航

登录后显示参考稿的品牌区和四个导航项；右侧保留当前用户名、账号报告入口所需状态和退出操作。窄屏时导航可横向滚动或折叠为可操作的紧凑布局，不能遮挡退出入口。

### 工作台

工作台使用当前会话任务计算四类事务的任务数和可解析工时占比，显示到期或逾期提醒，并提供“开始梳理”“每日跟踪”“历史记录”入口。没有任务时显示零状态，不填充参考稿的演示数据。

### 五步梳理流程

1. **事务填写**：沿用“昨天、今天、明天、后天”四栏输入。主操作先调用现有目标检查接口；检查通过后再调用任务提取接口并进入第二步，未通过时留在本步展示字段建议。
2. **AI 拆解确认**：使用参考稿表格展示现有任务字段。任务名、来源类别、截止时间、预估时长和重要/紧急状态均以可编辑控件呈现。无法可靠转换为日期或工时的值保留原文并标记“待确认”，不得猜测。
3. **时间分布诊断**：严格解析 `est` 的“小时/h/分钟”格式，转换为整数分钟；无法解析或非正数的任务不参与比例计算并列入待补全提示。四类占比按分钟计算到 0.1%，使用最大余数法保证展示总和为 100.0%。目标区间采用参考稿规则：昨天趋近 0%、今天 70–80%、明天 10–20%、后天约 5%。
4. **优先级排序**：调用现有矩阵判定接口，继续按稳定任务 ID 校验任务守恒，再以参考稿四象限布局展示。
5. **优化报告**：调用现有报告生成接口，继续安全渲染 Markdown、复制当前报告，并在报告可见后异步保存到账号历史。

步骤侧栏按现有有效状态锁定。用户可以返回已完成步骤；任何输入变化都必须使受影响的后续步骤重新计算。

### 每日跟踪

每日跟踪只读取当前会话任务。完成状态和实际完成时间存放在独立的前端 tracking 状态中，以任务 ID 关联，不把额外字段发送给现有 API。

- 勾选完成时写入浏览器当前时区的本地日期时间，可手动修正。
- 取消完成时移除对应完成时间。
- “结束当日并滚动”需要二次确认并展示将归档、滚动的数量。
- 已完成任务进入当前会话每日记录；未完成的“今天”任务在界面语义上滚入“昨天”。
- 同一任务在同一日不得重复归档；重复点击必须幂等。
- 刷新、退出或重新梳理后，tracking 状态和会话每日记录清空。

### 历史记录

历史页面使用参考稿的行式记录视觉，提供两个明确分区：

- **本次会话**：每日跟踪产生的临时记录，标注“刷新后清空”。
- **账号报告**：现有 SQLite 历史，标注“按账号保存”，保留分页、只读详情、复制、二次确认删除和失败提示。

二者不得混称为同一种持久化历史。

## 前端模块边界

当前 `frontend/app.js` 体积过大，此次改造按职责拆分但不改变服务端路由：

- `frontend/app.js`：应用启动、认证恢复、顶层路由和事件分发。
- `frontend/state.js`：会话状态、步骤门禁、下游失效和每日跟踪状态。
- `frontend/workflow-ui.js`：五步流程模板与水合逻辑。
- `frontend/dashboard-ui.js`：工作台指标、提醒和入口。
- `frontend/daily-tracking-ui.js`：完成登记、滚动和本次会话记录。
- `frontend/history-ui.js`：账号报告历史与历史页双分区。
- `frontend/time-distribution.js`：工时解析、分类映射、比例舍入和诊断状态。
- `frontend/styles.css`：参考稿视觉样式和响应式规则。

模块之间只通过显式参数和 `state.js` 导出的状态操作通信；UI 模块不直接调用数据库或浏览器持久化 API。

## 数据映射

现有服务端任务结构保持不变。前端按以下规则映射参考稿字段：

- `source=复盘` → 昨天。
- `source=今天` 或 `source=临时` → 今天。
- `source=短期目标` → 明天。
- `source=中长期` → 后天。
- 用户编辑类别时按反向规则写回：昨天 → `复盘`、今天 → `今天`、明天 → `短期目标`、后天 → `中长期`；不再把编辑后的类别写成 `临时`。
- `due` 仅在严格匹配 `YYYY-MM-DD` 时进入日期控件，否则作为“待确认”原文展示。
- `est` 仅接受正数加 `小时`、`h` 或整数加 `分钟`；转换后使用整数分钟参与计算。
- 用户编辑工时后，整小时或半小时按规范化的“小时”字符串写回，其他 15 分钟粒度按“分钟”字符串写回，以保持现有 API 契约。
- `importance`、`urgency` 和 `classificationSource` 继续沿用现有枚举和 AI 判定语义。

任何无法无损映射的数据都保留原文并提示用户确认，不能静默改写或虚构。

## 错误与异步状态

- 延续现有 AbortController 和请求序号机制，离开页面、重新梳理或发起新请求后，旧响应不得回写。
- 网络断开、模型超时、模型格式错误和历史保存失败均保留当前页面数据，并提供中文重试入口。
- 分布诊断遇到零总工时显示空状态，不生成 NaN、Infinity 或虚假百分比。
- 删除任务、结束当日和删除账号历史均采用目标明确的确认流程。
- 动态错误、成功提示和处理中状态使用 `aria-live`；处理中的按钮禁用并保留可读文案。

## 响应式与可访问性

- 桌面验收基准为 1280×800，窄屏验收基准为 375×812。
- 1000px 以下按参考稿切换为单栏；任务表头隐藏后，每个字段必须保留可见标签，不能只剩无语义输入框。
- 矩阵纵轴、长任务名、Markdown 表格和代码块不得导致整页横向滚动。
- 所有输入具有程序化 label，图标按钮具有可访问名称，弹窗支持 Escape 关闭和焦点回收。
- 尊重 `prefers-reduced-motion`，降低旋转和滚动动画。

## 测试与对抗式审查

实现采用测试先行：先添加失败用例，再写最小实现。

自动化覆盖：

- 参考稿关键页面结构、导航、五步标题和状态类。
- 工时解析、来源映射、零工时和 100% 精确舍入。
- 目标检查、任务提取、矩阵、报告四个 API 的真实请求顺序和数据贯穿。
- 编辑或删除后下游结果失效，旧异步响应不能污染新状态。
- 每日完成/取消、完成时间、单日幂等滚动和刷新清空。
- 本次会话记录与账号报告历史的存储边界。
- 已有账号历史保存、重试、分页、详情、删除和跨用户隔离回归。
- 1280×800 与 375×812 的无横向溢出、关键控件可见和步骤定位。
- 用户文本、Markdown、任务名和完成时间的注入防护。

对抗式审查额外尝试：重复快速点击、跨步骤返回、午夜边界、无效日期、异常工时、同名任务、空象限、超长任务名、100 条任务、登录状态变化、历史接口失败和浏览器不支持部分 API。审查结果写入 `docs/adversarial-review.md`，只关闭有自动化或可复现人工证据支持的问题。

## 验收标准

1. 甲方参考稿的主要页面和交互路径在真实项目中可复现，无演示种子数据和假登录。
2. 桌面与窄屏关键布局、组件层级、颜色和文案与参考稿一致，无明显错位或整页溢出。
3. 现有认证、四步 AI 接口、报告保存、账号隔离和安全 Markdown 回归全部通过。
4. 新增工作台、时间分布和每日跟踪不写入任何浏览器或服务端持久层。
5. 全量 Node 与 Playwright 测试通过，`git diff --check` 无错误。
6. 对抗式审查文档列出已验证项、发现的问题、修复证据和仍接受的风险。
