# 任务卡显式分类标签设计

## 目标

在“任务提取”页面的每张已分类任务卡上，同时展示重要性和紧急度的二值标签，避免用户把缺少标签误解为模型没有完成判断。

## 展示规则

| 原始字段 | 页面标签 | 样式 |
|---|---|---|
| `importance="高"` | 重要 | 保留现有紫色强调样式 |
| `importance="中"` 或 `"低"` | 不重要 | 使用现有中性灰色标签样式 |
| `urgency="高"` | 紧急 | 保留现有橙色强调样式 |
| `urgency="中"` 或 `"低"` | 不紧急 | 使用现有中性灰色标签样式 |
| `importance=null`、`urgency=null` | 不显示上述四种标签 | 只显示“待 AI 判定” |

中、低等级在四象限中都映射为“否”，因此任务卡使用“不重要”和“不紧急”表达二值矩阵含义；不把中、低改写成新的业务等级，也不改变后端数据。

## 修改范围

- 修改 `frontend/app.js` 的任务卡标签生成逻辑。
- 修改 `tests/frontend.spec.js`，覆盖重要且紧急、重要不紧急、紧急不重要、不重要不紧急，以及未分类手动任务。
- 不修改 API、数据库、提示词、矩阵算法、历史详情或报告。
- 不新增 CSS；负向标签复用现有中性 `.t` 样式，正向标签保留 `.imp` 和 `.urg`。

## 交互与异常边界

- 每张已分类任务卡始终有且仅有一个重要性标签和一个紧急度标签。
- 未分类手动任务不能提前显示“不重要”或“不紧急”，避免在 AI 判定前产生错误结论。
- 矩阵判定完成后，原本未分类的任务获得高/中/低等级，再按同一规则显示两个标签。

## 测试方法

1. 先补充 Playwright 失败断言，确认当前中/低等级任务缺少负向标签。
2. 最小修改 `taskTags()`，确认定向测试通过。
3. 运行完整 Playwright 和项目测试，检查现有任务流程、矩阵和历史功能无回归。
4. 运行 `git diff --check` 和 Git hook，检查最终 Git 状态。

## 验收标准

- 已分类任务明确显示“重要/不重要”和“紧急/不紧急”。
- 正向标签使用现有强调色，负向标签使用中性样式。
- 未分类手动任务只显示“待 AI 判定”。
- 四象限映射、55/25/15/5、任务 ID 和其他正式契约保持不变。
