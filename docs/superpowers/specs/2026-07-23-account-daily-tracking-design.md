# 账号每日跟踪持久化设计

## 背景与目标

当前“每日跟踪”只使用浏览器当前会话中的任务和完成状态。用户退出、刷新或重新登录后，必须重新执行五步流程，无法持续维护当天清单。

本次改造将“每日跟踪”升级为按账号、按自然日持久化的统一清单：

- 每个账号每天只有一份每日跟踪清单。
- 清单汇总该账号当天生成的所有历史记录中的全部任务。
- 用户可以从任意历史详情进入每日跟踪，但页面始终打开当前日期的清单。
- 用户对清单的编辑、完成状态和删除操作自动保存。
- 历史记录继续作为原始报告快照，不随每日跟踪变化。

“当天”统一按 `Asia/Shanghai` 时区的自然日判断，以历史记录的服务端 `created_at` 为准；任务自身的截止时间、来源类别和“昨天/今天/明天/后天”不影响是否进入当天清单。

## 范围

### 本期包含

- 新增账号每日跟踪持久化表、仓储和受保护接口。
- 汇总当前账号当天生成的所有历史记录任务。
- 按稳定的任务内部 `task.id` 去重。
- 保存任务字段编辑、完成状态、完成时间、展示顺序和删除结果。
- 历史详情始终提供“进入每日跟踪”入口。
- 约 800ms 无新操作后的防抖自动保存。
- 保存状态、失败重试、跨标签页冲突和跨日保护。
- 数据迁移、服务端测试和浏览器回归测试。

### 本期不包含

- 不按任务名称去重。
- 不把旧历史记录的任务导入今天清单。
- 不修改原始历史任务、矩阵或报告。
- 不新增手动“保存”按钮。
- 不新增跨日期浏览或编辑历史每日清单的入口。
- 不新增日历排期、通知提醒、团队共享或实时多人协作。
- 不改变现有五步 AI 工作流和历史记录生成契约。

## 核心产品规则

1. 一个账号在一个自然日只有一份持续使用的每日跟踪清单。
2. 今日清单的来源是该账号今日生成的全部历史记录，不是用户刚才查看的某一条历史记录。
3. 用户从旧历史记录点击“进入每日跟踪”时，仍只进入今天清单；旧记录任务不参与合并。
4. 同一 `task.id` 在今日多个历史记录中重复出现时只保留一项；名称相同但 ID 不同的任务全部保留。
5. 已保存的每日编辑优先于历史快照中的同 ID 原始字段。
6. 今日后来生成的新历史记录产生新任务 ID 时，新任务追加到现有清单末尾，不覆盖已保存内容。
7. 删除任务需二次确认。确认后记录其 ID，后续加载不得从历史记录自动补回。
8. 每日清单编辑只影响每日跟踪数据；历史详情永远展示生成报告时的原始快照。

## 数据模型

新增非破坏性 migration（迁移），创建 `daily_tracking_days` 表，不改写 `time_management_runs`：

| 字段 | 用途 |
| --- | --- |
| `id` | 每日清单 UUID |
| `user_id` | 所属账号，外键关联用户 |
| `tracking_date` | `YYYY-MM-DD` 格式的上海时区业务日期 |
| `tasks_json` | 当前可见任务的可编辑快照及顺序 |
| `tracking_json` | 按任务 ID 保存的完成状态和完成时间 |
| `removed_task_ids_json` | 已确认删除的来源任务 ID 集合 |
| `revision` | 乐观并发控制版本号 |
| `created_at` | 创建时间 |
| `updated_at` | 最近自动保存时间 |

约束和索引：

- `UNIQUE(user_id, tracking_date)` 保证同账号同一天只有一份清单。
- `user_id` 删除时级联清理该账号的每日清单。
- JSON 字段由仓储层统一校验和序列化，损坏数据不得直接进入前端。
- `revision` 每次成功写入加一，用于阻止旧标签页覆盖新数据。

`tasks_json` 保存每日跟踪副本，不复用或回写历史记录中的 `tasks_json`。`tracking_json` 与任务内容分离，避免完成状态污染现有任务契约。

## 服务端组件边界

### 日期与来源查询

新增统一业务日期工具，按 `Asia/Shanghai` 计算当天起止时间。历史仓储新增“按用户和创建日期查询”能力，只返回当前账号、当天生成的记录。

日期过滤必须在服务端完成，不能信任客户端提交的用户 ID、时区或日期范围。

### 每日跟踪仓储

仓储负责：

- 按 `user_id + tracking_date` 读取清单。
- 创建或更新每日清单。
- 校验任务 ID、完成状态、删除 ID 和 JSON 结构。
- 在事务中比较并递增 `revision`。
- 保证账号隔离和同日唯一性。

### 合并服务

合并逻辑使用任务 ID 映射，规则如下：

1. 收集当天全部历史记录任务，并按历史生成顺序及任务原顺序展开。
2. 相同 `task.id` 只接收第一次出现的来源任务。
3. 已保存且未删除的任务保留每日清单中的编辑字段、完成状态和顺序。
4. `removed_task_ids_json` 中的任务不重新加入。
5. 今日来源中尚未出现过的新 ID 追加到清单末尾。
6. 每次新增来源任务后触发一次自动保存，使合并结果成为持续清单的一部分。

任何缺少合法稳定 ID 的来源任务均视为数据错误：接口保留已有每日清单，返回可诊断错误，不使用任务名称临时生成 ID。

## API 设计

### 读取今日清单

`GET /api/time-management/daily-tracking/today`

响应包含：

- `trackingDate`
- `tasks`
- `tracking`
- `revision`
- `updatedAt`
- `sourceSummary`：当天历史记录数量和合并后任务数量
- `hasUnpersistedMerge`：是否合并了需要自动保存的新来源任务

服务器只使用当前认证用户和上海时区当天日期，不接受客户端指定其他账号或日期。

### 自动保存今日清单

`PUT /api/time-management/daily-tracking/today`

请求包含：

- `trackingDate`
- `tasks`
- `tracking`
- `removedTaskIds`
- `revision`

写入规则：

- 使用当前登录用户，不接受请求体中的 `user_id`。
- 校验 `trackingDate` 仍是当前上海日期；跨日请求返回冲突。
- 校验 CSRF（跨站请求伪造）令牌和同源请求。
- 比较 `revision`；版本不一致返回 `409 Conflict`。
- 保存前重新查询当天来源任务，把请求发出期间产生的新任务安全追加。
- 成功后返回新 `revision`、`updatedAt` 和服务器最终清单。

现有历史读取、删除和五步工作流接口保持不变。

## 前端交互

### 历史详情入口

账号历史详情新增“进入每日跟踪”按钮，并对所有日期的历史记录显示。点击只执行页面导航，不把当前历史记录作为导入参数。

每日跟踪页标题下明确显示当前日期和来源摘要，例如：“已汇总今天生成的 2 条记录，共 11 项任务”，避免用户误解为正在跟踪刚才打开的旧记录。

当天没有历史记录或任务时显示空状态，引导用户先完成五步流程，不填充演示任务。

### 编辑与自动保存

用户可以：

- 编辑现有任务字段。
- 勾选或取消完成。
- 修改实际完成时间。
- 二次确认后删除任务。

任一变更先立即更新页面状态，再启动约 800ms 防抖计时。计时期间继续操作会重置计时，停止操作后调用保存接口。页面不提供手动“保存”按钮。

页面显示四种保存状态：

- `已自动保存`
- `正在保存…`
- `有未保存更改`
- `保存失败，点击重试`

自动保存成功后以服务器响应替换本地 `revision` 和规范化结果。失败时保留全部本地编辑，不回滚为服务器旧数据。

### 离开保护

- 已自动保存时，正常导航、退出和关闭页面不弹提示。
- 存在尚未发送、正在保存或保存失败的更改时，页面内导航和退出登录弹出明确确认。
- 浏览器刷新或关闭使用原生 `beforeunload` 提示。
- 用户取消离开后继续保留编辑内容并可重试保存。

## 冲突、跨日与失败处理

### 多标签页冲突

每次保存携带最近读取的 `revision`。如果另一标签页已经成功保存，旧标签页收到 `409 Conflict`，不得自动覆盖。

页面提示“每日清单已在其他页面更新”，保留当前本地草稿，并允许用户重新加载服务器版本。首期不实现字段级自动合并，避免产生不可预测的覆盖。

### 跨日保护

页面加载后保存固定携带当时的 `trackingDate`。如果操作跨过上海时区零点，服务器拒绝把前一天草稿写入今天清单，前端提示重新进入今日跟踪。

### 网络和数据错误

- 读取失败：显示错误和重试入口，不展示伪造空清单。
- 保存失败：保留本地草稿及失败状态，允许重试。
- 来源任务结构异常：返回带 `requestId` 的安全错误，历史数据和既有每日清单均不改动。
- 未认证、会话过期或 CSRF 失败：沿用现有认证错误处理，不将用户输入记录到日志。
- 所有服务端日志只记录请求 ID、路由、状态和耗时，不记录任务正文。

## 安全与数据完整性

- 所有每日跟踪接口必须先认证，仓储查询必须包含当前 `user_id`。
- 写接口继续执行现有同源与 CSRF 防护。
- 前端渲染任务文本时继续使用安全文本或现有转义机制。
- 请求体限制任务数量、字段长度和 JSON 总大小，避免异常负载。
- 历史记录为只读来源；每日跟踪更新不得调用历史更新路径。
- migration 只新增表，可通过停止新代码并删除空的新表回退；回退前若已有每日数据，应先备份数据库。

## 测试策略

实现采用测试先行：先增加失败测试，再编写最小实现。

### 服务端测试

- 同一用户同一天只能创建一份清单。
- 不同用户的数据完全隔离。
- 上海时区自然日边界和跨零点保存拒绝。
- 当天多条历史记录的全部任务合并。
- 相同 ID 去重，同名不同 ID 保留。
- 已保存编辑和完成状态优先于来源快照。
- 新来源任务追加且不覆盖已有编辑。
- 已删除 ID 不再补回。
- `revision` 冲突返回 `409`，旧版本不写入。
- 身份认证、CSRF、输入长度和异常 JSON 校验。
- 每日保存不改变 `time_management_runs`。

### 浏览器回归测试

- 从当天及旧历史详情点击按钮均进入今天清单。
- 页面显示当前日期和正确来源摘要。
- 编辑、完成、修改时间停止约 800ms 后自动保存。
- 自动保存状态按请求结果正确变化。
- 刷新和重新登录后恢复最新清单。
- 删除确认后任务消失，重新进入不恢复。
- 保存完成后离开不提示；待保存或失败时离开提示。
- 保存失败保留输入并可重试。
- 多标签页冲突和跨日冲突不静默覆盖。
- 当天无记录时显示真实空状态。
- 现有五步流程、历史详情、复制和删除功能继续通过。

## 验收标准

1. 每个账号每日只有一份持久清单，并汇总该账号当天生成的所有历史记录任务。
2. 从任意历史详情进入时都只打开今天清单，不导入旧记录。
3. 相同任务 ID 去重；同名不同 ID 保留。
4. 编辑、完成状态、完成时间和删除操作自动保存，刷新或重新登录后仍保持。
5. 当天后来生成的新任务自动追加，不覆盖已保存修改。
6. 删除并保存的任务不会再次出现。
7. 原始历史记录内容始终不变。
8. 保存失败、并发冲突和跨日操作都有明确提示，不发生静默数据丢失。
9. 账号隔离、认证、CSRF 和输入校验回归通过。
10. 数据库迁移不改写现有历史表，全量相关 Node 与 Playwright 测试通过，`git diff --check` 无错误。
