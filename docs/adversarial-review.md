# 时间管理助手 · 对抗式审查

审查日期：2026-07-16

## 结论

**生产结论：Request changes。** 当前材料可以作为视觉与提示词设计基线，但还不是可用智能体。前端没有模型/API与真实任务状态，任务提取、矩阵和报告仍是固定样例。

独立 v2 提示词与测试用例已设为权威材料；PDCA、SMART 和矩阵方法内嵌于提示词，不另建重复知识库。原始外置 HTML 与 ZIP 内版本 SHA-256 均为 `b66163360da2874247fcf5c92e4a3065a93c1e7e64b3a315d09ece628e74a296`。

## 本次已修复

- 未完成目标检查时不能继续进入任务提取。
- 空输入的“补全”改为待填写模板，不再把获客、投诉等演示事实写成用户事实。
- 手动任务的耗时字段改用安全 DOM 文本节点，已用 HTML 注入回归测试验证。
- 重要/紧急矩阵按左“不紧急”、右“紧急”正确排列。
- “复制报告”现在真实写入剪贴板，并有可清理临时节点的失败回退。
- 四栏输入在返回时保留；编辑会使检查失效；离开、重开或处理动画中返回会取消旧检查与跳转定时器。
- 以上行为由 8 个 Playwright 用例覆盖。

## 阻断项

### Critical：核心流程不消费用户输入

`runAI()` 只有动画与定时跳页。四栏输入、删除和手动新增任务都不会影响后续固定任务、矩阵与报告。必须建立单一状态树和四步真实调用链，并在任务增删后重新计算下游结果。

### High：当前“AI 检查”仍不是 PDCA/SMART 审查

本次只修复了绕过和虚构事实；静态原型仍主要依赖字数判断，不能证明昨天具备 PDCA 四环，也不能证明短期目标满足 SMART。真实接入时必须消费 v2 步骤 1 的结构化 JSON，并展示具体缺项。

### High：矩阵比例契约在数学上不可验收

提示词要求返回四个区间字符串，同时要求合计约 100%。四区间下界合计 86%，上界合计 125%，区间本身无法求和；空象限也不应占固定比例。

建议输出 `recommended_range` 与数值 `allocated_pct`，空象限为 0，四个 `allocated_pct` 严格合计 100；或明确这些仅是政策区间，不宣称已经计算分配。

### High：任务状态与真实请求生命周期仍缺失

本次已为四栏目标补上状态保留、完整 reset、检查/跳转令牌和旧回调防回写；但任务列表仍只存在于 DOM，删除和手动新增在离开页面后会丢失，也不会影响硬编码矩阵与报告。接入真实 API 时仍需把任务、矩阵、报告和请求状态纳入同一状态树，并使用真正的请求取消机制。

### High：日期、工时和容量模型不足

“昨天/明天/后天”同时被用作季度、本月、年度语义；提示词没有参考日期、时区、工作日历或可用工时。耗时是任意字符串，截止日期也没有过去日期与容量校验。真实契约应使用 `referenceDate`、`timezone`、分钟数/区间和可用分钟数。

### High：提示注入与输出验证缺失

四栏、任务 JSON 和矩阵 JSON 会跨步骤直接拼接。必须把用户与模型内容作为不可信数据隔离，实施长度/数量限制、JSON Schema、枚举、任务 ID、任务守恒和输出上限。合法 JSON 但非法语义同样必须拒绝。

### High：规范漂移

- 旧合并稿缺少 v2 的 PDCA/SMART 规则，已归档但不得接入。
- “跟进客户投诉”来自“今天”输入，测试与前端却标为“临时”，破坏来源溯源。
- 提示词任务输出没有截止时间，前端却要求并展示截止时间。
- v2 输出示例中后三个 `status` 仍是空字符串，不是完整枚举示例。

应从一个版本化 Schema 生成前端类型、提示词输出示例和断言。

### High：隐私治理缺失

输入可能包含客户投诉、绩效和人员日程。接入 API 前需明确脱敏、访问权限、供应商数据用途、日志策略、保留期限和删除方式，默认不记录原始提示正文。

### High：模型测试仍不可执行

`tests/prompt-cases.md` 是叙述式检查表，缺少完整输入、精确 JSON、自动断言、模型版本、重复运行稳定性与日期基准。当前 8 个 Playwright 用例只证明本次前端修复，不证明模型决策正确。

## 其他重要问题

- 重要性/紧急度为高、中、低，但矩阵只定义“高→是”，中与低如何映射不清。
- 宽泛目标要求“拆到可执行”又要求“不新增”，应把推导结果标为待用户确认的候选任务。
- 已完成的“昨天”事项可能被误提取为未来任务，需要 `status` 与只提取未完成行动的规则。
- 下游只传任务名，没有稳定 ID，同名任务无法可靠删除、去重或追踪。
- 可点击步骤、label 关联和动态提示缺少完整无障碍语义。

## 必须补充的对抗测试

1. 空白、超长、Unicode/控制字符、直接与间接提示注入、HTML/脚本字符串。
2. 非 JSON、非法枚举、重复/同名任务、任务丢失或跨象限重复。
3. 零/负数/超过 24 小时/区间工时、过去截止日、月末、闰日、跨时区。
4. 无任务、单任务、千条任务、所有任务同象限、总工时超容量。
5. 前进后退数据保留、重新开始清空、并发点击、旧请求回写。
6. 删除/新增任务后矩阵同步，以及四个数值分配严格合计 100。

## 进入真实开发的验收门

- 建立真实状态树、版本化 Schema 与四步 API 调用。
- 修正矩阵、日期、工时和可用容量模型。
- 安全 Markdown/DOM 渲染边界已落地；生产接入仍需补齐提示注入防护、JSON Schema 校验与隐私约束。
- 把提示词用例变成可执行契约回归，并与现有前端 E2E 一起持续运行。
